# 目录合并引擎改造 — Proposal

## Why

当前目录生成 (`catalog.py`) 存在三个根因问题：

### 问题 1：格式要求被碎片化使用

`format_requirements` 已从源文档第3章提取了 **26项必须提交的材料**，但目录生成器只对其中6个做了零散的 `is_format_required()` 关键词检查。剩余20项（承诺函、基本情况表、人员情况表等）被静默忽略。

### 问题 2：评分维度→响应章节映射断裂

评分标准中的主观评审项（服务方案40分、售后保障方案20分）共占 **60分**，但没有自动生成对应的撰写章节。原因是 `doc_has()` 只检查章节目录标题关键词，而源文档章节标题（如"第五章比选项目及要求"）不含"服务方案"等关键词。

### 问题 3：数据管线不一致导致评分章节为空

`save_review()` 覆盖 `analysis["scoring"]` 时使用了 check-items 的数据格式（`{business, technical}`），但 `_build_scoring_section()` 读取的是原格式（`{dimensions: [...]}`）。两端格式不一致，导致评分章节子项为空。

### 问题 4：子项硬编码而非数据驱动

商务响应、技术响应等章节的子项是硬编码字符串，未从实际解析的 `business.items` / `technical.items` 动态生成。

### 问题 5：评分标准和废标条件被误放入目录

当前有"评分标准响应"章节（空children），但评分标准是评审专家用的打分表，不是投标文件的响应内容。废标条件是撰写人自查清单，也不应作为目录章节。

## What

设计一套 **多源合并引擎**，统一管理目录生成的数据流：

```
输入数据源：
  - format_requirements（第3章格式要求） → 主骨架（最高优先级）
  - scoring.dimensions（第6章评分维度）  → 补充/映射章节
  - business/technical items（第5章要求）→ 填充子项描述
  - check_items（用户确认）              → 填充资格/符合性
  - document_chapters                   → 完整性验证

核心逻辑：
  合并算法（4阶段）：
    阶段1：解析 format_requirements 为树（9大章+17子项）
    阶段2：评分维度→
      - 客观项（报价、业绩）→ 标记已覆盖
      - 主观项（服务方案、售后）→ 新增章节
      - 评分标准原文 → 不进目录，作为内容生成参考
      - 废标条件 → 不进目录，作为合规约束
    阶段3：business/technical items 动态填充子项
    阶段4：document_chapters 覆盖完整性验证
```
