# 目录生成改造：让目录依赖前序步骤的输出

## Why

当前目录生成（`catalog.py`）有三个断层，导致生成的目录结构不合理、不可用：

### 断层 1：忽略已选择的包号

分析阶段检测到所有包的数据存入 `analysis_data`，但目录生成时读取的是**未经包过滤的原始全量 JSON**。每个包的技术参数、商务要求、评分标准可能完全不同，当前生成的目录描述可能是"多个包的混合体"。

证据：`catalog.py` 中 `selected_package_no` 引用次数为 **0**。

### 断层 2：忽略核对确认项

用户在核对阶段逐项确认了资格要求、实质性要求、废标条件等，这些确认状态存入 `BiddingCheckItem` 表。但目录生成时**完全不读取确认状态**，导致：
- 用户已通过的资格项没有作为章节展开
- 标记为 `attention` 的项没有在目录中提示
- 用户修改过的内容没有反映到目录描述

证据：`catalog.py` 中 `BiddingCheckItem` 引用次数为 **0**。

### 断层 3：硬编码 3 章结构

```python
title_profiles = {
    "GOODS": {
        "requirement": ["项目概述与采购范围", "技术参数响应", "商务资质与评分应答"],
    },
}
return {"outline": [3 个标题]}
```

无论项目复杂程度、产品数量、评分维度多少，永远只生成 3 个顶级章节。但真实场景中：

| 项目类型 | 合理章节数 |
|---|---|
| 简单货物采购（1-2 个产品） | 6-8 章 |
| 复杂检测试剂采购（26 个产品） | 9-12 章 |
| 服务类项目 | 8-10 章 |
| 工程项目 | 10-15 章 |

每个项目生成的目录应该不一样——因为选择了不同包、确认了不同项、产品数量和评分维度不同。

## 目标

1. `selected_package_no` → 目录只生成当前包相关的章节
2. `BiddingCheckItem` 确认状态 → 已确认的资格项自动展开为章节
3. 包内 `items` + `scoring.dimensions` → 动态决定章节数量和颗粒度
4. 兼容性：无包号/无确认项时自动回退到当前 3 章结构
5. 前端零改动（目录 JSON 格式不变）

|---|---|---|
