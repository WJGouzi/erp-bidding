# 目录生成改造 — Proposal

## Why：当前系统在三个层面上有病

### 层面 1：数据可信度未知（置信度缺失）
```
当前系统：
  提取到了 → 写入值
  没提取到 → 写入空字符串 ""
  
  下游无法区分「文档没有这个信息」vs「提取逻辑漏了」
  
  举证：
  - metadata.project_code = "" → 不知道是比选编号不存在还是 regex 没匹配到
  - business_requirements = "暂未提取到商务要求。" → 同样无法区分
  - 用户看到空数据但不确定是否可靠
```

### 层面 2：数据契约不一致（schema 混乱）
```
- _build_metadata 初始化 project_code = ""（字符串）
- KEY_MAP 期望 project_code = {"value": "CG20250099"}（字典）
- 下游 reader 又期望是字符串
- 三方各说各话，值被静默丢弃

类似的断裂点在全系统有 10+ 处
```

### 层面 3：格式要求被忽略（无结构化约束）
```
- 招标文件的"第三章 比选申请文件格式"是强制性规范
- 定义了响应函/报价表/授权书等固定模板
- 当前系统把这些内容等同于普通段落
- Chroma 语义召回不保证命中 → LLM 自由发挥 → 格式不符 = 废标
```

## What：三个维度的改造

### 维度 A：置信度系统（解决"数据可信度"）
- 每个提取字段附带 confidence (0~1) + source + raw_match
- 校验门在分析完成后强制执行最低置信度阈值
- 低置信度字段在目录/check-items 中标注"待确认"

### 维度 B：数据契约（解决"schema 混乱"）
- 中央 Schema：所有模块围绕同一个数据定义编程
- @property 视图淘汰 legacy 副本字段
- 类型不匹配阻断流程（不再静默丢弃）

### 维度 C：格式要求管线化（解决"格式要求被忽略"）
- Phase 1.5 新增格式要求提取模块
- 目录生成时校验必选章节完整性
- 内容生成时注入模板表格作为硬约束

## How：实施步骤

### Step 1：打地基（P0）
- 定义 Central Schema
- 实现 Validation Gate
- 所有提取模块输出带置信度

### Step 2：修 bug（P0）
- 修复已发现的 10 个具体问题
- 消除 legacy 副本字段

### Step 3：建新能力（P1）
- Phase 1.5 格式要求提取
- Catalog/生成集成格式约束

## 风险

| 风险 | 概率 | 影响 | 缓解 |
|---|---|---|---|
| Schema 迁移影响现有数据 | 高 | 中 | _safe_read 兼容新旧格式 |
| 置信度阈值不好设 | 中 | 低 | 先宽松后收紧，持续调优 |
| 格式提取准确率不够 | 中 | 中 | 先提取结构再落实到内容 |
| 改动量大周期长 | 高 | 高 | 分 Step 交付，Step 1/2 先行 |
