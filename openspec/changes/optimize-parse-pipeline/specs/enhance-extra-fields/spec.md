# 商务字段提取率提升

## 目标
提升 `metadata.extra` 中 `payment_terms`、`delivery_location`、`service_period`、`pricing_rule` 等商务字段的提取率，减少"暂未提取到商务要求"的情形。

## 影响文件
- `app/service_modules/task_pipeline/analysis_v3/phase1_metadata.py` — `_rule_extract()` 和 `_build_metadata()`

## 设计原则

### 正则添加规范

1. **不引用具体标书的原文作为正则**，而是基于中文商务条款的通用表述模式
2. 每个字段的正则应覆盖主流表述变体，但不过度匹配
3. 增加"交叉验证"：同一信息在不同位置出现时，用置信度择优

### 冗余处理

同一字段可能出现在多个位置（封面、前附表、正文、表格），多个正则匹配到同一字段时：
- 数值型：取最大值（如保证金比例）
- 文本型：取最长匹配（最完整的信息）
- 特殊标记（如"据实结算"）：优先于数值，标记 note

### 字段与标书类型无关性

extra 字段的提取逻辑对所有标书类型统一——不因文档类型不同而切换正则。同一字段在不同采购方式下的不同表述已在正则层面覆盖。

## 改动内容

### 1. 现有正则扩展：覆盖更多中文表述变体

以下正则变体全部基于中文商务条款的常见表述，不从任何具体标书原文推导：

```python
# ── 交付地点（当前仅"配送地点""交货地点""交付地点""供货地点"）
# 扩展变体：合并表达（"服务期限及配送地点"）、省略表达（"地点：xxx"）
("delivery_location", r"(?:.*?地点|地址)[：:]\s*(.+?)(?:。|；|\n{2,}|$)"),
# 注意：此正则要求上下文中有"地点""地址"关键词，避免与"开标地点""报名地点"混淆
# 后续可在 _build_metadata 中与 bid_submission_location 做去重

# ── 服务期限（当前仅"服务期限""交货期限""交货期""供货期""工期"）
# 扩展变体：合并表达、"服务期"、"合同期"
("service_period", r"(?:服务期|合同期|质保期)[：:]*[^。]*?(\d+)\s*(?:年|月|天|日)"),

# ── 验收标准（当前偏弱，仅匹配"财库""国标""GB""参照"开头的描述）
# 扩展变体：更通用的验收表述
("acceptance_standard", r"(?:验收|考核|检验)[：:]*\s*(.+?)(?:。|；|\n{2,}|$)"),

# ── 付款方式（当前仅"付款方法""付款方式""支付方式""付款条件"）
# 扩展变体：已在 _rule_extract 中覆盖主流变体
("payment_terms", r"(?:付款方法|付款方式|支付方式|付款条件|结算方式)[：:]\s*(.+?)(?:。|；|\n{2,}|\n(?:[^\n]*[：:]|$))"),

# ── 报价方式（当前已覆盖"一次性报价""多轮报价""据实结算""固定单价""固定总价"）
# 扩展变体：增加"综合单价""包干价"
("pricing_rule", r"(?:报价|本次报价|报价方式)[^。]*?(一次性报价|多轮报价|据实结算|固定单价|固定总价|综合单价|包干价|按实结算)"),
("pricing_rule", r"金额[：:]\s*(据实结算|按实结算)"),
```

### 2. `_build_metadata()` 的回退推断

```python
# 如果价格相关字段包含"据实结算"标记，同步更新 pricing_rule
# 这不算项目特定——"据实结算"是政府采购中常见的预算表述
if extra.get("pricing_rule") or budget_note_contains("据实结算"):
    extra["pricing_rule"] = extra.get("pricing_rule") or "据实结算"
```

### 3. 去重与择优

```python
# 在 _build_metadata() 中，对 extra 字段做最终清理：
# 1. delivery_location 如果与 bid_submission_location 相同 → 保留两者
# 2. 合并表达中提取的字段（如"服务期限及配送地点"）→ 两个字段都赋值
```

## 验收标准
1. 写明了"付款方式"的标书 → extra.payment_terms 有值
2. 写明了"配送地点/交货地点/供货地点"的标书 → extra.delivery_location 有值
3. "据实结算"类表述 → extra.pricing_rule 有值且 budget.note 有标注
4. 合并表达（"服务期限及配送地点"）→ 两个字段均提取
5. 不引入新的误匹配（如将"开标地点"误认为 delivery_location）

## 不包含
- 不改动 `analysis_data` 的顶层结构
- 不增加新的 extra 字段
- 不改变 `business_requirements` 的拼接逻辑
- 不从具体标书文档提取项目特定的正则模式
