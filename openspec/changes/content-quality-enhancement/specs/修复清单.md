## 一、P0 — 严重问题

### 1.1 封面标的名称/编号为空

**现象**:
```
标的名称：           ← 空
项目编号：           ← 空
投标人名称：erp      ← 显示"erp"而非公司全称
```

**根因**:
1. `_build_docx_bytes()` 中 `cover_item_name` 从 `bidder_notice.get("project_name")` 取值，但分析阶段可能未提取到此字段
2. 投标人名称直接从 `subject_context.get("company_name", "")` 取值，但该字段存的是系统注册名/简称"erp"，而非主体表中的公司全称
3. `SubjectCompany.company_name` 可能是正确的全称，但 `_build_subject_material_context()` 返回的 `company_name` 来源需核实

**已做修复**:
- `_build_docx_bytes()` cover 部分已增加多级兜底：`bidder_notice` → `analysis_result` 顶层 → `analysis_data` JSON
- `_extract_analysis_context()` 中已增加从 `bidder_notice` 提取 "标的名称"/"项目编号" 的兜底

**仍需修复**:
- [ ] 核实 `SubjectCompany.company_name` 的实际值，确保 `_build_subject_material_context()` 返回正确全称
- [ ] 在 `_build_subject_material_context()` 中增加日志输出，确认返回的 company_name

### 1.2 XXX 占位符未替换（含下划线格式）

**现象**:
```
"比选申请文件正本1份，副本XXX份。"
"本单位erp参加______的比选活动"
"XXX天（日历日）"
"XXX（项目名称）"
```

**根因**:
1. `_identify_placeholders_via_llm()` 的 LLM prompt 可能未正确识别占位符，或 LLM 服务不可用导致降级失败
2. 占位符有两种格式：`XXX（说明文字）` 和 `______`（下划线），LLM prompt 可能未覆盖下划线格式
3. `_fill_template()` 的字段映射：LLM 识别结果中 `field_hint` 未正确映射到 `_TEMPLATE_FIELD_MAP` 中的字段

**修复方案**:
- [ ] `_identify_placeholders_via_llm()` 的 prompt 增加下划线 `____` 格式识别
- [ ] 增加正则兜底：当 LLM 不可用或返回空时，用正则 `r'XXX（[^）]+）'` 和 `r'_{2,}'` 提取占位符
- [ ] `_identify_placeholders_via_llm()` 增加详细的日志输出，便于调试
- [ ] `_fill_template()` 在替换后增加验证：检查结果中是否仍有未替换的 `XXX` 或 `____`

**涉及函数**:
| 函数 | 位置 | 修改内容 |
|------|------|---------|
| `_identify_placeholders_via_llm()` | helpers.py:1870 | 增加下划线格式识别+正则兜底 |
| `_fill_template()` | helpers.py:1923 | 增加替换后验证 |
| `_TEMPLATE_FIELD_MAP` | helpers.py:1750 | 增加更多 hint 同义词 |

### 1.3 报价一览表产品数据为空

**现象**:
```
┌──────┬──────────────────────┬────────┬──────┬────────┬────────┬──────┐
│ 序号 │ 标的名称             │ 规格型号│ 数量 │ 单价   │ 总价   │ 备注 │
├──────┼──────────────────────┼────────┼──────┼────────┼────────┼──────┤
│ 1    │ 详见技术参数要求      │        │ 1    │        │        │      │
└──────┴──────────────────────┴────────┴──────┴────────┴────────┴──────┘
```
期望：从招标文件表格中提取的具体产品清单，包含所有品目的名称、规格、数量、限价。

**根因**:
1. `analysis_data.table_classification.product_lists` 中有结构化产品数据（采购产品名称、★规格参数、★数量等）
2. 旧代码 `_extract_table_data_from_analysis()` 对"报价一览表"只从 `technical_requirements` 文本行解析，不使用 `product_lists`
3. `_build_product_context()` 只从 `effective_text` 用窄正则 `(采购|供货|产品)[：:：\s]*([^\s，。,\.]{2,30})` 提取产品词，不读 `product_lists`

**已做修复**:
- [x] `_build_product_context()` 新增从 `analysis_data.table_classification.product_lists` 提取产品名称
- [x] `_extract_analysis_context()` 新增传递 `_raw_product_lists` 到 context
- [x] `_extract_table_data_from_analysis()` 报价表分支优先从 `_raw_product_lists` 取数据

**仍需修复**:
- [ ] 重新生成验证修复后的报价表填充结果
- [ ] 若产品名包含特殊字符（★、※等），确保清洗逻辑正确处理

### 1.4 主体信息表数据全错

**现象**:
| 字段 | 当前值 | 期望值 |
|------|--------|--------|
| 单位名称 | `erp` | 公司全称（如"成都XXX科技有限公司"）|
| 注册地址 | `12312312312` | 实际地址 |
| 统一社会信用代码 | 空 | 18位信用代码 |
| 法定代表人 | 空 | 法人姓名 |
| 成立时间/注册资本 | 空 | 有值 |
| 联系人 | 空 | 有值 |
| 联系电话 | 空 | 有值 |

**根因**:
1. `_build_subject_material_context()` 中 `company_name = subject.company_name or ""` — 但 `SubjectCompany.company_name` 存的是系统注册名"erp"，而非用于投标的公司全称
2. 表中其他字段（`address`, `contact_person`, `contact_phone` 等）在 context 中存在，但 `_extract_table_data_from_analysis()` 的"基本情况表"分支只取了 `company_name`, `address`, `credit_code`, `contact_person`, `contact_phone`，context 中缺少 `address` 等字段

**修复方案**:
- [ ] 检查 `SubjectCompany` 表的实际数据，确认 `company_name`、`address`、`credit_code` 等字段的值
- [ ] 若 `company_name` 存的是简称，则应在主体管理页面增加"投标用公司全称"字段，或确保用户录入全称
- [ ] `_build_subject_material_context()` 增加 `address`、`contact_person`、`contact_phone`、`legal_person` 字段（当前只返回了 `company_name` 和 `credit_code`）
- [ ] `_extract_table_data_from_analysis()` 的"基本情况表"分支使用正确的 context key

## 二、P1 — 中等问题

### 2.1 承诺函未被正确填充

**现象**:
```
本单位erp参加______的比选活动，现承诺我单位满足以下要求：
（一）具有独立承担民事责任的能力；
（二）具有良好的商业信誉和健全的财务会计制度；
```
期望：
```
本单位成都XXX科技有限公司参加CG20250099项目的比选活动，现承诺...
```

**根因**:
1. 占位符格式为 `______`（下划线）而非 `XXX（比选申请人名称）`，`_identify_placeholders_via_llm()` 未识别
2. `company_name` 为 "erp" 而非全称，即使替换成功也是错误值

**修复方案**:
同上 1.2（占位符识别）和 1.4（公司全称）

### 2.2 主体资料附件无实际内容

**现象**: 文档末尾列出标题（财务报表、廉洁承诺书、营业执照等），但无图片/文本内容。

**根因**:
1. `_write_material_block()` 调用 `_load_material_visual_payloads()` 读取文件，但可能因 MinIO/Storage 连接失败或文件路径错误导致无内容
2. `_write_formatted_content(document, text_excerpt)` 中 `text_excerpt` 为空（因置信度过滤被清空）

**修复方案**:
- [ ] 在 `_write_material_block()` 中增加详细日志，输出每个材料的 `file_id`、`storage_provider`、读取结果
- [ ] 检查 `StorageService.read_bytes()` 是否能正常读取文件（MinIO 连接是否正常）
- [ ] `_build_subject_material_context()` 中置信度过滤已改为仅打标签不清除文本（本轮已修复），重新生成后应能看到文本摘录

### 2.3 投标人名称使用系统名

**现象**: 全文多处显示 "erp" 作为公司名称。

**根因**: 与 1.4 相同，`SubjectCompany.company_name` 存的是系统注册名。

**修复方案**: 同 1.4。

## 三、P2 — 较低问题

### 3.1 正文内容重复

**现象**: 比选函等章节内容在文档中重复出现2-3次（段落 33-39、89-96、110-112）。

**根因**:
`_split_generated_sections_by_titles()` 或 `_extract_child_content_sections()` 中，当生成的内容包含与标题相同的文字时，内容拆分逻辑可能出错，导致同一段文本被分配到多个章节。

**修复方案**:
- [ ] `_split_generated_sections_by_titles()` 中去重：确保每个章节的内容只被分配一次
- [ ] 增加日志输出 `chapter_title → content length`，便于排查

### 3.2 业绩表/人员表完全空白

**现象**: Table 4（业绩一览表）仅1行表头无数据；Table 6（人员情况表）仅1行表头无数据。

**根因**:
这两个表的数据需要从主体资料或用户手动录入，系统无数据源。当前代码对"业绩"表只返回空行。

**修复方案**:
- [ ] 对空表插入占位提示文字："请根据实际情况填写"
- [ ] 如果主体资料中有相关材料（如过往合同扫描件），尝试 OCR 提取业绩信息

### 3.3 表数据行填充不完整

**现象**: 偏离表、应答表等只有 1 行泛泛的 "完全响应"，而非逐条响应招标要求。

**根因**:
`_extract_table_data_from_analysis()` 的"偏离表/应答表"分支只从 `technical_requirements` 文本行中按 `/^\d+[.、]/` 正则提取，但 `technical_requirements` 文本可能未包含完整的逐条要求。

**修复方案**:
- [ ] 增加从 `analysis_data.table_classification.tech_requirements` / `business_requirements` 的结构化数据中提取条目的逻辑
