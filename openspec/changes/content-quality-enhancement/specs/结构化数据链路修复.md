# 结构化数据链路修复

> 将 analysis_data JSON 中的结构化数据完整传递到 LLM prompt

---

## 修复内容

### 1. `_extract_analysis_context()` 新增提取

| 字段 | 数据结构 | 用途 |
|------|---------|------|
| `_eligibility` | `{qualifications[], starred_requirements[], disqualifications[]}` | 资格要求、实质性要求、废标条件 |
| `_table_classification` | `{product_lists[], tech_requirements[], business_requirements[]}` | 产品清单、技术/商务要求表 |
| `_format_requirements` | 模板格式信息 | 招标文件规定的格式要求 |
| `_scoring` | `{method, total_score, dimensions[]}` | 评分标准结构化 |
| `_packages` | `[{parameters: {core_products[], ...}}]` | 包参数、核心产品 |

### 2. LLM Prompt 新增7个结构化区块

插入在原有的 `结构化商务要求`/`结构化技术要求` 文本块之前，按以下顺序提供给 LLM：

```
[结构化] 资格要求清单（逐项）           ← eligibility.qualifications[]
[结构化] ★ 实质性要求（必须完全响应）    ← eligibility.starred_requirements[]
[结构化] 废标条件（不可违反）             ← eligibility.disqualifications[]
[结构化] 产品清单表                      ← table_classification.product_lists[]
[结构化] 技术参数要求（逐项）             ← table_classification.tech_requirements[]
[结构化] 商务要求（逐项）                 ← table_classification.business_requirements[]
[结构化] 评分维度                        ← scoring.dimensions[]
[结构化] 核心产品列表                     ← packages[].parameters.core_products[]
```

每个区块保留原始结构化信息（逐条列举，含材料要求、规格参数等），而非合并为文本块。

### 3. 向下兼容

原有的 TEXT 字段（`technical_requirements`, `business_requirements` 等）仍保留作为回退。
