# 数据递进查找与链路修复

> 基于 2026-07-02 全链路代码审查和原则讨论

---

## 核心原则

| # | 原则 | 说明 |
|---|------|------|
| P1 | **主体优先** | 主体信息表已有数据直接使用，不做置信度校验 |
| P2 | **三级递进** | 主体→知识库→留白一页，逐级查找，找不到就空着 |
| P3 | **表格原样** | 招标文件的表格原样复制到生成文档，在此基础上填充 |
| P4 | **纯LLM识别** | 占位符识别只用LLM，不用正则，正则仅做位置修正 |
| P5 | **固定格式不降级** | 承诺函等固定模板填不了就留空，不走LLM改写 |
| P6 | **全量表格内容** | 报价一览表完整复制，产品信息从产品库匹配填充 |

---

## 数据递进查找模型

```
┌─────────────────────────────────────────────────────────────┐
│                     Step 1: 主体信息表                       │
│  SubjectCompany (company_name, credit_code, address...)     │
│  SubjectMaterialFile (营业执照, 法人证, 财务报表...)          │
│  → 直接读取，0 置信度校验                                   │
│  → text_excerpt = _read_file_text()[:800]                   │
└──────────────────────────┬──────────────────────────────────┘
                           │ 找到 → 填充
                           │ 没找到 ↓
┌──────────────────────────┴──────────────────────────────────┐
│                     Step 2: 知识库                           │
│  ChromaDB MultiRecallEngine                                 │
│  → 多路召回 + score ≥ 0.3                                   │
│  → 仅对知识库片段做置信度门控                                │
└──────────────────────────┬──────────────────────────────────┘
                           │ 找到 → 填充
                           │ 没找到 ↓
┌──────────────────────────┴──────────────────────────────────┐
│                     Step 3: 留白一页                         │
│  → 插入空白页 (分页符 + 章节标题 + "本节无内容")           │
│  → 下个章节从下一页开始                                      │
│  → 日志记录 "XX项未找到匹配资料"                             │
└─────────────────────────────────────────────────────────────┘
```

### 涉及函数改造

| 函数 | 当前行为 | 目标行为 |
|------|---------|---------|
| `_generate_qualification_content()` | 只查 subject_context，没有 knowledge_contexts 参数 | 接收 knowledge_contexts，实现三级递进 |
| `_build_leaf_response_bindings()` | evidence 只从 analysis_context + subject 取 | 增加知识库 evidence 路径 |
| `_EMPTY_PAGE_MARKER` | 文本标记 `[[EMPTY_PAGE]]` | 分页符 + 章节标题 + 空白说明 |
| `_build_docx_bytes()` | 遇到 `[[EMPTY_PAGE]]` 跳过或写空行 | 插入分页符 |

## 产品列表修复规范

### 数据流

```
招标文件表格 (.docx)
      │
      ▼
┌──────────────────────────────────────────────────┐
│  分析阶段: 两个并行解析系统                        │
│                                                    │
│  系统A: table_parser.py                           │
│    parse_all_tables() → table_results             │
│    → product_lists[]: items[{name, spec, qty...}] │
│                                                    │
│  系统B: table_classifier.py                       │
│    classify_all_tables() → _classification        │
│    → product_lists[]: items[{采购产品名称,       │
│       规格参数, 数量...}]                         │
└────────────────────┬─────────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────────┐
│  生成阶段: 必须统一字段名映射                      │
│                                                    │
│  当前问题:                                         │
│  系统A 输出 field: "name"                         │
│  系统B 输出 field: "采购产品名称"                  │
│  _extract_table_data_from_analysis() 硬编码       │
│  item.get("采购产品名称", "") → 不兼容系统A       │
│                                                    │
│  修复: 统一到 _PRODUCT_HEADER_MAP 映射             │
└────────────────────┬─────────────────────────────┘
                     │
                     ▼
┌──────────────────────────────────────────────────┐
│  表格原样复制到生成文档                            │
│                                                    │
│  Step 1: 从 analysis_data.table_classification    │
│     中读取原始表格结构 (headers + rows)            │
│                                                    │
│  Step 2: 在 docx 中重建相同表格                   │
│     - 表头行: 保持原标题和列数                     │
│     - 数据行: 完整复制                             │
│                                                    │
│  Step 3: 从产品库匹配填充空余信息                  │
│     - 对每个产品的 name 做 LLM 语义检索            │
│     - 匹配产品库中最相似的产品信息                 │
│     - 填充规格参数、品牌、单价等空列               │
│                                                    │
│  Step 4: 找不到的列保持空白                        │
└────────────────────────────────────────────────────┘
```

def _match_products_from_library(product_items):

## 占位符识别规范 (纯 LLM)

### 不再使用的函数

| 函数 | 原因 |
|------|------|
| `_fallback_extract_placeholders()` | 不再使用正则兜底 |
| `_FALLBACK_PLACEHOLDER_PATTERNS` | 不再使用模式匹配 |

### 保留但仅做位置修正

| 函数 | 新用途 |
|------|--------|
| `re.finditer` 匹配位置 | 仅用于修正 LLM 返回的 start/end 偏移量，不用于兜底提取 |

### LLM Prompt 设计

```python
prompt = """你是一个占位符识别助手。
找出下面文本中所有需要填写的空白位置。

规则：
1. 只识别，不填充，不改写原文
2. 返回 JSON 数组格式
3. 每个元素包含：raw(占位符原文), start(起始字符位置), end(结束位置), hint(推测字段含义)

识别所有格式的占位符，包括但不限于：
- XXX（字段名）格式：XXX（比选申请人名称）
- 下划线格式：______
- 混合格式：法定代表人：__________
- 隐式空白：比选日期：  年   月   日
- 方括号格式：【】
- 尖括号格式：<>
- 任何看起来需要填写的空白位置

如果无法推断 hint，用 "unknown"。
如果文本中没有占位符，返回空数组 []。

文本：
{template_text}

只返回 JSON 数组："""
```
