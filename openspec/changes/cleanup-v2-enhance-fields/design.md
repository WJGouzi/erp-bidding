## Context

### Current Architecture

当前 v3 管线执行顺序：
```
上传文件 → document_parser (DOCX→StructuredDocument)
        → start_analyze_v3:
            1. phase1_metadata: regex 扫描 raw_text 提取元数据
            2. phase2_eligibility: 章节文本扫描资格/废标/★条款
            3. phase3_scoring: 评分表解析 + 分包参数统计
            4. _build_strategy: 纯规则策略
            5. assemble_v3_analysis_data → 序列化到 analysis_data JSON
```

关键局限性：
1. **表格数据只通过 flatten 后的文本接触** — 原生表格的结构化信息（行/列对应关系、合并单元格）完全丢失
2. **包分析只在文本层面** — 未按包切分内容，所有包共享同一套评分/资格分析结果
3. **无文档类型识别** — 比选/招标/竞争性谈判使用同样的正则集

### Constraints
- 纯规则驱动，零 LLM 调用（维持现有架构原则）
- 不引入新的外部依赖
- 向后兼容：现有 `analysis_data` schema 增加字段而非破坏性修改
- 保持解析速度（目标：单文档 < 5s）

### Stakeholders
- **前端用户**：希望在 UI 上看到每包独立信息、表格数据的结构化展示
- **后端开发者**：希望 JSON schema 可读性高、字段含义清晰
- **标书撰写专家**：希望分析结果反映专家思维，而非机械的关键词匹配

---

## Goals / Non-Goals

### Goals
1. **表格解析增强**：识别标书中 5 类表格（前附表/评分表/产品清单/资质表/响应格式表），从中提取结构化数据
2. **分包独立分析**：每个包独立执行资格扫描、参数统计、策略分析，输出差异化策略建议
3. **文档分类驱动**：自动识别采购方式（招标/比选/竞争性谈判等），加载对应解析规则
4. **元数据增强**：增加 confidence/source 标注，extra 字段补全
5. **零 LLM 保持**：所有新增功能纯规则驱动

### Non-Goals
- PDF 解析优化（独立问题）
- ChromaDB 入库优化（独立问题）
- 前端展示改动（只改后端 JSON 输出）
- 非标书类文档的解析（如合同模板、中标公告）

## Decisions

### Decision 1: 表格解析器设计 — 类型识别优先于通用解析

**选择**：三阶段流水线：类型识别 → 按类型策略提取 → 融合入 metadata

```
阶段1：类型识别
  输入 → docx table object (rows x cols)
  输出 → table_type 枚举

  判定规则（按优先级）：
  ┌──────────────────────────────────────────────────────────┐
  │ 3列 + 表头含"内容""说明与要求"        → PRELIMINARY_TABLE│
  │ 5+列 + 表头含"评分""得分""分值"       → SCORING_TABLE    │
  │ 5+列 + 表头含"品名""规格""品牌"       → PRODUCT_LIST     │
  │ 表头含"供应商""注册地址""法定代表人"   → QUALIFICATION_TBL│
  │ 表头含"比选文件条目号""响应文件的应答" → RESPONSE_FORMAT  │
  │ 其他                                   → GENERIC_TABLE    │
  └──────────────────────────────────────────────────────────┘

阶段2：按类型策略提取
  - PRELIMINARY_TABLE → key-value 提取
    处理逻辑：
      - 固定 3 列格式（序号|内容|说明与要求）
      - "内容"列文本作为 key，"说明与要求"列文本作为 value
      - 多行 value 合并（如"成交供应商数量"的复杂描述）
      - 特殊字段解析：数字提取、枚举值识别

  - SCORING_TABLE → 增强评分维度提取
    现有 parse_scoring_table 基础之上扩展：
      - 识别子维度（如"报价"下的"参与报价20分"+"供货20分"）
      - 保存评分标准原文
      - 检测评分方法行

  - PRODUCT_LIST → 结构化产品数据
    处理逻辑：
      - 识别表头映射（品名→name, 规格→spec, 数量→qty）
      - 合并跨列标题（常见于 DOCX 合并单元格）
      - 输出按包归类

阶段3：融合
  - 各类型表格提取的结果合并到 metadata
  - 与 regex 结果冲突时，表格提取优先（结构置信度高）
  - 在 metadata 中标记 `_source: "table"` 或 `_source: "text"`
```

**为什么不是通用 NLP/ML 表格解析**：
- 标书表格的格式高度结构化（固定列数、固定位置）
- 规则驱动的准确率可达 95%+，且零推理成本
- 表格类型有限（≤6种），无需训练模型

**为什么不是直接增强 regex**：
- regex 无法处理带合并单元格的复杂表格
- regex 破坏行列对应关系（flatten 后丢失"第5行第2列"的上下文）
- 原生 DOCX 表格对象可以直接访问 cell-by-cell

### Decision 2: 分包分析 — "按包切分"架构

**选择**：分析管线中插入包切分步骤，逐包执行资格/参数/策略分析

```
当前流程：
  文档 → 整体资格扫描 → 整体评分提取 → 包号检测 → 策略(同质化)

新流程：
  文档 → 文档分类 → 表格解析 → 包检测
       ├→ 公共部分：资格扫描(三包共用条款)
       ├→ 包1：资格(包专用) → 参数统计 → 策略(差异)
       ├→ 包2：资格(包专用) → 参数统计 → 策略(差异)
       └→ 包3：资格(包专用) → 参数统计 → 策略(差异)
       └→ 包间关联分析
```

包内容切分策略：
```
包的边界检测（按优先级）：
  1. 章节标题含"第X包"、"包X" → 直接映射
  2. 表格内标注"采购包X" → 该表格归入对应的包
  3. 段落含"(采购包X)"或"（包X）" → 该段落归入对应的包
  4. 未匹配内容 → 归入"公共"部分

按包归集后：
  - 对每包独立计算：★/▲条款数、核心产品、资格要求
  - 公共部分的资格条件复制到每包（如"三包共用"标注）
  - 评分维度按包标注（如某评分项只适用于包2/3）
```

**为什么不是保持整体分析+包后处理**：
- 后处理无法区分"这个资格条件是包专用还是公共" 
- 策略分析需要每包独立的输入才能输出差异化的建议
- 包切分在前端展示中也更自然（用户选择包→看到该包完整分析）

### Decision 3: 文档分类 — 轻量关键词分类器

**选择**：基于规则的三层判定，无需 ML 模型

```
第一层：文件名/首段判定
  - 文件名含"比选" → bid_type: SELECTION
  - 文件名含"竞争性谈判" → bid_type: NEGOTIATION
  - 文件名含"询价" → bid_type: INQUIRY
  - 默认 → bid_type: TENDER

第二层：正文关键词确认
  - "比选公告"、"比选文件" → confirm SELECTION
  - "竞争性谈判公告" → confirm NEGOTIATION
  - "招标公告"、"招标文件" → confirm TENDER
  - 置信度计算公式：匹配关键词数 / 候选关键词总数

第三层：规则集加载
  - 根据确认的 bid_type 加载对应规则集
  - TENDER 用词：采购人/招标代理/投标保证金
  - SELECTION 用词：比选人/比选代理/比选保证金/综合评分法
  - NEGOTIATION 用词：采购人/谈判小组/最终报价

规则集继承：每种类型从通用规则集继承，覆盖差异字段
  通用规则 → TENDER 规则 → 覆盖 purchaser/agent 为"采购人/招标代理"
  通用规则 → SELECTION 规则 → 覆盖为"比选人/比选代理"
```

**为什么不是 LLM 分类**：
- 文档类型判定是一个简单的 NLP 任务，规则即可达到 99% 准确率
- 分类在解析管线的最前端，引入 LLM 会阻塞后续流程
- 规则失败时可降级为默认 TENDER 类型

### Decision 4: 元数据 Confidence 标注

**选择**：对每个提取的字段附加 confidence 和 source 信息，不破坏现有 schema

```json
// 当前格式
{
  "project_name": "德阳市疾病预防控制中心项目",
  "budget": { "total": 0, "packages": {} }
}

// 新格式（向后兼容）
{
  "project_name": "德阳市疾病预防控制中心项目",
  "project_name_meta": {
    "value": "德阳市疾病预防控制中心项目",
    "confidence": "high",
    "source": "text:regex",
    "matched_pattern": "项目名称：(.+?)"
  },
  "budget": {
    "total": 0,
    "note": "全额赔付",
    "packages": {},
    "_meta": {
      "total_confidence": "high",
      "total_source": "table:preliminary",
      "note_confidence": "high",
      "note_source": "text:regex"
    }
  },
  "document_type": {
    "value": "SELECTION",
    "confidence": "high",
    "source": "filename:contain '比选' + text:contain '比选公告'"
  }
}
```

## Risks / Trade-offs

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| 表格类型误判（如评分表被判为产品清单） | 中 | 中 | 多特征联合判定（列数+表头关键词+数据类型），可降级为 GENERIC |
| 包切分边界不准（复杂文档中包内容交叉引用） | 中 | 高 | 保守策略：无法确定归属的内容放入"公共"，人工可干预 |
| 文档类型识别错误（如"比选"文件用了"招标"词表） | 低 | 中 | 第二层正文关键词确认为兜底；降级为 TENDER 通用模式 |
| 解析性能下降（表格解析+逐包分析增加耗时） | 低 | 低 | 表格解析 O(n_rows)，包分析并行，预期增量 < 500ms |
| 向后兼容破坏（前端依赖现有字段） | 低 | 高 | 所有新增字段加 `_meta` suffix，不修改现有字段类型和含义 |

## Migration Plan

1. **Phase 1 — 表格解析模块**（2天）
   - 创建 `table_parser.py`：类型识别 + 前附表解析 + 产品清单解析
   - 集成到 `phase1_metadata.py`：表格提取结果优先于 regex
   - 单元测试：3 份真实标书表格验证

2. **Phase 2 — 分包独立分析**（2天）
   - 重构 `phase3_scoring.py`：包切分 → 逐包分析 → 包间关联
   - 更新 `schemas.py`：每包差异化的策略输出
   - 集成测试：德阳疾控（3包）验证

3. **Phase 3 — 文档分类**（1天）
   - 新增 `_classify_document()` 分类函数
   - 规则集按类型加载
   - `metadata` schema 增加 `document_type` 字段

4. **Phase 4 — 元数据增强**（1天）
   - `schemas.py` 增加 confidence/source 标注
   - extra 字段补全
   - 整体回归测试

**回滚策略**：每个 Phase 单独可回滚；新增字段不影响现有数据读取

## Open Questions

1. 产品清单（如德阳疾控的 1200+ 种试剂）是否需要在 analysis_data 中完整保存？还是只汇总统计（总品种数、分类统计）？
2. 包间关联分析中"哪个包利润高/竞争少"的判定，是否需要一个基础的行业数据库（如各品类平均利润率）？
3. confidence 字段是否需要暴露到前端 UI，还是仅供后端调试使用？
